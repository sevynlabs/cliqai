    **1. Create CalendarService** at `apps/api/src/modules/scheduling/calendar/calendar.service.ts`:
    Injectable service with ConfigService for GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GOOGLE_REDIRECT_URI env vars.

    - `getOAuth2Client()` — Creates `new google.auth.OAuth2(clientId, clientSecret, redirectUri)`. Returns the client.
    - `getAuthUrl(organizationId: string, professionalId?: string)` — Generates OAuth URL with `access_type: 'offline'`, scope `['https://www.googleapis.com/auth/calendar']`, `prompt: 'consent'` (CRITICAL — forces refresh_token every time), state encoded as `${organizationId}:${professionalId || 'default'}`.
    - `exchangeCode(code: string)` — Calls `oauth2Client.getToken(code)`, returns `{ refreshToken: tokens.refresh_token }`. Throw if no refresh_token.
    - `saveToken(organizationId: string, professionalId: string | null, calendarId: string, refreshToken: string)` — Upserts CalendarToken via Prisma. Refresh token stored as-is for now (encrypt with pgcrypto in production — add TODO comment referencing LGPD-04).
    - `getCalendarClient(refreshToken: string)` — Creates OAuth2 client, calls `setCredentials({ refresh_token: refreshToken })`, returns `google.calendar({ version: 'v3', auth: oauth2Client })`.
    - `getTokenForOrg(organizationId: string, professionalId?: string)` — Finds CalendarToken from DB. Returns token or null.

    **2. Create CalendarController** at `apps/api/src/modules/scheduling/calendar/calendar.controller.ts`:
    Controller `@Controller('calendar')`:
    - GET /calendar/auth?professionalId= — Returns { authUrl } from CalendarService.getAuthUrl. RBAC: owner, admin.
    - GET /calendar/callback?code=&state= — Exchanges code, saves token. Decodes state to get orgId/professionalId. Lists calendars via `calendar.calendarList.list()` and saves the primary calendarId. Redirects to dashboard settings page.

    **3. Create AvailabilityService** at `apps/api/src/modules/scheduling/availability/availability.service.ts`:
    Injectable service with CalendarService, PrismaService, ConfigService.

    - `checkFreeBusy(calendarId: string, refreshToken: string, startISO: string, endISO: string)` — Calls `calendar.freebusy.query()`. CRITICAL: Check `result.data.calendars[calendarId]?.errors` array before trusting busy data (Pitfall 4). Returns busy ranges array.
    - `getAvailableSlots(organizationId: string, procedureName?: string, date?: string)` — 
      1. Load CalendarToken for org (default professional for now)
      2. Load AgentConfig for operating hours (operatingHoursStart, operatingHoursEnd, timezone)
      3. Generate time range: if date provided use that date, otherwise use next 3 business days
      4. Call checkFreeBusy for the range
      5. Calculate free slots: start from operatingHoursStart, slot duration = 60min default (or procedure-specific if procedureName matches a known procedure), buffer time = 15min default (SCHED-05)
      6. Filter out slots that overlap busy ranges
      7. Filter out slots outside operating hours (SCHED-07)
      8. Return array of `{ start: string, end: string, formatted: string }` where `formatted` is PT-BR human-readable (e.g., "Quarta, 3 de abril as 14:00")
      Use `date-fns` (install if not in api package: `pnpm --filter @cliniq/api add date-fns`) for date arithmetic.

    **4. Create AppointmentsService** at `apps/api/src/modules/scheduling/appointments/appointments.service.ts`:
    Injectable service with CalendarService, AvailabilityService, RedisService, PrismaService.

    - `bookSlot(params: { organizationId, leadId, calendarId, refreshToken, startISO, endISO, procedureName, patientName, patientPhone, timezone })` —
      1. Acquire Redis lock: `booking:lock:${calendarId}:${startISO}` with SET NX PX 30000 (30s TTL). If fails, throw ConflictException('Horario sendo reservado por outra sessao').
      2. Re-check freeBusy under lock (race condition guard — SCHED-06)
      3. Insert TENTATIVE event via `calendar.events.insert()` with summary `[Cliniq] ${procedureName} - ${patientName}`, description with phone/procedure, status: 'tentative', timezone.
      4. Create Appointment record in Prisma: organizationId, leadId, calendarEventId (from event.data.id), calendarId, startAt, endAt, status: 'tentative', procedureName.
      5. Update Lead stage to 'agendado' via direct Prisma update.
      6. Finally block: release Redis lock using Lua script (atomic check-then-delete to avoid deleting another thread's lock).
      Return the Appointment record.

    - `confirmAppointment(appointmentId: string, organizationId: string)` — Load appointment, update Google Calendar event status to 'confirmed', update Appointment.status to 'confirmed', update Lead stage to 'confirmado'.
    - `cancelAppointment(appointmentId: string, organizationId: string)` — Delete Google Calendar event, update Appointment.status to 'cancelled'. Do NOT change lead stage (attendant decides).
    - `rescheduleAppointment(appointmentId: string, organizationId: string, newStartISO: string, newEndISO: string)` — Cancel old event, book new slot (reuses bookSlot logic).
    - `findByLead(leadId: string)` — Returns appointments for a lead.
    - `findUpcoming(organizationId: string, days?: number)` — Returns upcoming appointments for org.

    **5. Create AppointmentsController** at `apps/api/src/modules/scheduling/appointments/appointments.controller.ts`:
    Controller `@Controller('appointments')`:
    - GET /appointments — findUpcoming with optional ?days= param
    - GET /appointments/lead/:leadId — findByLead
    - POST /appointments/:id/confirm — confirmAppointment
    - POST /appointments/:id/cancel — cancelAppointment
    - POST /appointments/:id/reschedule — rescheduleAppointment with body { startISO, endISO }
    RBAC: owner, admin, manager, attendant.

    **6. Create SchedulingModule**: Imports PrismaModule, RedisModule. Providers: CalendarService, AvailabilityService, AppointmentsService. Controllers: CalendarController, AppointmentsController. Exports: CalendarService, AvailabilityService, AppointmentsService.

    **7. Update app.module.ts**: Add SchedulingModule to imports. Add GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GOOGLE_REDIRECT_URI to env schema validation.

    **1. Create schedule.node.ts** at `apps/api/src/modules/agent/graph/nodes/schedule.node.ts`:
    Export `createScheduleNode(availabilityService: AvailabilityService, appointmentsService: AppointmentsService, calendarService: CalendarService, prisma: PrismaService)` factory function.

    The schedule node handles the `qualificationStage === 'schedule'` path. It uses an LLM call to determine the lead's intent from the last message:

    - **If lead is asking about availability / wants to schedule**: Call `availabilityService.getAvailableSlots(state.organizationId, state.procedureInterest)`. Format slots into a PT-BR message offering 3-5 options. Return message with slots.

    - **If lead is confirming a specific slot** (e.g., "quero o horario das 14:00"): Parse the selected slot from the message (LLM classifier with temperature 0, extract startISO/endISO from available slots context). Load CalendarToken for org. Call `appointmentsService.bookSlot(...)` with leadId from state. On success: return message confirming the appointment in PT-BR with date/time. Set `appointmentId` in state. On ConflictException: return message saying slot was taken, offer alternatives.

    - **If lead wants to cancel**: Call `appointmentsService.cancelAppointment(state.appointmentId, state.organizationId)`. Return cancellation confirmation message.

    - **If lead wants to reschedule**: Call cancel then present new availability.

    LLM intent classification: Use ChatAnthropic (max 30 tokens, temperature 0) to classify intent as 'check_availability' | 'confirm_slot' | 'cancel' | 'reschedule' | 'other'. If 'other', return a gentle PT-BR message asking what they'd like to do about scheduling.

    The node returns `{ messages: [new AIMessage(reply)], appointmentId }`.
