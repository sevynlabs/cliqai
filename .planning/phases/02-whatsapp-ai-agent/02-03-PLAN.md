1. **ethics-guard.node.ts**: Pre-response classifier that checks the AI's draft response BEFORE it's sent.
   - Input: state with messages (last message is the AI's draft response from qualify/greet).
   - Blocked phrase list (PT-BR): "diagnóstico", "diagnostico", "garanto que", "resultado garantido", "cura", "100% de sucesso", "certeza de resultado", "prometo", "sem risco", "totalmente seguro". Also regex patterns: /\d+%\s*(de\s*)?(sucesso|chance|resultado)/, /vai\s*(curar|resolver|eliminar)/.
   - First: Check draft against blocked phrases (string matching + regex). If match found, rewrite the message by calling ChatAnthropic with a short system prompt: "Rewrite this message removing any medical claims, diagnosis, or result promises. Keep the helpful tone. Original: {draft}". Temperature: 0.3. Max tokens: 200.
   - Second (if first check is clean): Run a fast LLM classifier call to ChatAnthropic with: "Does this message contain medical diagnosis, treatment promises, or result guarantees? Reply SAFE or BLOCKED followed by reason." Temperature: 0. Max tokens: 50. If BLOCKED, rewrite.
   - Return: Updated messages array with the cleaned/original AI message. Add a flag `ethicsBlocked: boolean` in state for logging (add this field to SDRState in this plan).
   
   IMPORTANT: The dual check (phrase blocklist THEN LLM classifier) is intentional. The blocklist catches obvious violations instantly (no LLM cost). The LLM classifier catches subtle/creative violations the blocklist misses. Both must pass.

2. **operating-hours.node.ts**: Timezone-aware operating hours enforcement.
   - Input: state with agentConfig (has operatingHoursStart, operatingHoursEnd, timezone).
   - Use `Intl.DateTimeFormat` with `timeZone` option to get current hour in clinic's timezone (no external library needed — Node.js native). Extract hour from `new Date().toLocaleString('en-US', { timeZone: config.timezone, hour12: false, hour: 'numeric' })`.
   - If current hour is OUTSIDE operatingHoursStart..operatingHoursEnd:
     - Generate a polite PT-BR message: "Olá! Nosso horario de atendimento e das {start}h as {end}h. Retornaremos assim que possivel! {personaName}" (use agentConfig.personaName).
     - Return: `{ messages: [new AIMessage(deferralMessage)], qualificationStage: 'consent' }` — stays at current stage, doesn't advance. The reply is sent, conversation paused until next business hours.
   - If INSIDE hours: Return empty update (pass through).
   
   NOTE: Do NOT use date-fns-tz or luxon. Intl.DateTimeFormat is built into Node.js 18+ and handles timezone conversion natively.

3. **emergency-detect.node.ts**: Emergency keyword detection on the USER's incoming message.
   - Input: state.messages — check the LAST human message (not AI message).
   - Emergency keywords (PT-BR, case insensitive): "emergencia", "emergência", "urgente", "sangramento", "sangue", "desmaio", "desmaiou", "infarto", "avc", "acidente", "hospital", "samu", "192", "pronto socorro", "dor forte", "dor intensa", "não consigo respirar", "alergia grave", "inchaço", "reação alérgica".
   - If ANY keyword detected: Return `{ humanHandoffRequested: true, qualificationStage: 'handoff' }` + add an AI message: "Detectei uma situacao de urgencia. Estou transferindo voce imediatamente para nossa equipe. Se for uma emergencia medica, por favor ligue 192 (SAMU) ou va ao pronto-socorro mais proximo."
   - If no emergency: Return empty update (pass through).

4. **handle-objection.node.ts**: Handles common lead objections with empathetic responses.
   - Input: state with messages (last human message) and agentConfig.
   - Objection detection: Call ChatAnthropic with a classifier prompt: "Is this message an objection? Categories: price (caro, não tenho, desconto), fear (medo, dor, anestesia, risco), timing (depois, agora não, vou pensar), info_request (manda informação, envia foto, portfolio). Reply: NONE or CATEGORY." Temperature: 0. Max tokens: 20.
   - If objection detected: Call ChatAnthropic with a tailored system prompt per category:
     - price: Acknowledge concern, mention flexible payment options, redirect to value/benefits.
     - fear: Empathize, explain modern techniques are comfortable, offer to answer specific concerns.
     - timing: Respect their timing, mention limited availability, suggest a no-commitment consultation.
     - info_request: Acknowledge, promise to send info, but first ask one qualifying question to keep conversation going.
   - Use agentConfig.tone and agentConfig.personaName in the response. Max tokens: 250. Temperature: 0.7.
   - Return: `{ messages: [aiMessage], qualificationStage: 'qualify', consecutiveUnresolved: 0 }` — routes back to qualify after handling.
   - If no objection: Return empty update with qualificationStage unchanged (graph routing will skip this node).

   - `ethicsBlocked`: Annotation&lt;boolean&gt; (default false) — tracks if ethics guard rewrote the last response.

2. **loop-guard.node.ts**: Detects conversation loops and max turns.
   - Input: state with turnCount, consecutiveUnresolved, agentConfig.maxTurns.
   - Check 1: `turnCount >= agentConfig.maxTurns` (default 20). If exceeded → return `{ humanHandoffRequested: true, qualificationStage: 'handoff' }`.
   - Check 2: `consecutiveUnresolved >= 3`. Three turns in a row where qualify node extracted zero new BANT info → conversation is stuck. Return `{ humanHandoffRequested: true, qualificationStage: 'handoff' }`.
   - If neither condition met: Return empty update (pass through).

3. **human-handoff.node.ts**: Executes when human handoff is triggered.
   - Input: state with organizationId, remoteJid, instanceName.
   - Generate handoff message (PT-BR): "Vou transferir voce para um de nossos especialistas que podera te ajudar melhor. Aguarde um momento! {personaName}"
   - Update Conversation in DB: set status to 'human_handling'.
   - Enqueue notification to 'notifications' BullMQ queue: `{ type: 'handoff_required', organizationId, remoteJid, reason: determineReason(state) }`. Reason can be: 'max_turns', 'loop_detected', 'emergency', 'user_requested'.
   - Return: `{ messages: [aiMessage], humanHandoffRequested: true, qualificationStage: 'handoff' }`.

4. **sdr.graph.ts**: Rewire the FULL graph with all nodes. Replace the Plan 02-02 graph definition:
   ```
   Nodes: consent_check, operating_hours, emergency_detect, greet, qualify, handle_objection, ethics_guard, loop_guard, human_handoff
   
   Edges:
   START -> consent_check
   consent_check -> conditional:
     - consentGiven=true -> operating_hours
     - else -> END (waiting for consent)
   
   operating_hours -> conditional:
     - outside hours -> END (deferral message sent by node)
     - inside hours -> emergency_detect
   
   emergency_detect -> conditional:
     - emergency detected -> human_handoff
     - no emergency -> route_by_stage (conditional based on qualificationStage)
   
   route_by_stage:
     - 'greet' -> greet
     - 'qualify' -> qualify
     - 'objection' -> handle_objection
     - 'handoff' -> human_handoff
     - 'schedule' -> END (Phase 3)
   
   greet -> ethics_guard
   qualify -> loop_guard
   handle_objection -> ethics_guard
   
   loop_guard -> conditional:
     - humanHandoffRequested -> human_handoff
     - else -> ethics_guard
   
   ethics_guard -> END (message ready to send)
   human_handoff -> END
   ```

   This means EVERY AI response passes through ethics_guard before being sent. Loop guard checks happen after every qualify turn. Operating hours and emergency are checked on every inbound message before any other processing.
